# Architecture

`idomarketing` is a **single-deployment Laravel monolith** serving an Inertia single-page app. There is no separate REST API — server routes return `Inertia::render('page')` and the matching React component renders client-side.

> This page is the practical overview. For the full pattern, dependency rules, and code examples, see [`.ai-factory/ARCHITECTURE.md`](../.ai-factory/ARCHITECTURE.md).

## High-Level Overview

```
Browser (React 19 SPA)
    │  Inertia visits (XHR)
    ▼
Laravel routes ──► Controller ──► Service ──► Repository ──► Eloquent / SQLite
    │                                 │
    │  Inertia::render()              └─► Rich model (domain logic)
    ▼
React page component (resources/js/pages/**)
```

## Directory Structure

```text
app/
├── Modules/            # Feature modules (new work) — controllers/services/repositories/models
├── Actions/Fortify/    # Custom auth actions (create user, reset password)
├── Concerns/           # Shared validation-rule traits
├── Http/
│   ├── Controllers/    # Base + settings controllers
│   └── Middleware/      # HandleInertiaRequests, HandleAppearance
├── Models/             # Framework-level models (User)
└── Providers/          # App + Fortify service providers (composition root)

routes/                 # web.php, settings.php, console.php (auth routes come from Fortify)
resources/js/
├── pages/              # Inertia page components (map 1:1 to Inertia::render strings)
├── components/  ui/    # React components + shadcn-style primitives
├── layouts/            # App / auth / settings shells
└── routes/  actions/   # GENERATED by Wayfinder — never hand-edit
tests/                  # Pest: Feature/ (primary) + Unit/
```

## Architecture Pattern: Structured Modules

New feature areas (e.g. campaigns, audiences, analytics) are built as **modules** under `app/Modules/<Module>/`, each with its own technical layers:

| Layer | Responsibility |
|-------|----------------|
| `Controllers/` | Thin presentation — validate (Form Requests), call a service, render/redirect |
| `Services/` | Application services — orchestrate a use case; **no** business rules |
| `Repositories/` | Data access behind an interface (Eloquent implementation) |
| `Models/` | **Rich** Eloquent models — business rules and invariants live here |

**Dependency direction (strictly downward):**

```
Controller → Service → Repository (interface) → Model → Database
```

- ✅ Depend on repository **interfaces**, bound to Eloquent implementations in a Service Provider.
- ❌ No layer skipping (controllers must not query Eloquent directly).
- ❌ No upward dependencies (models never import services/controllers).
- ❌ No cross-module reach-in — call another module's public Service or emit an event.

Existing auth/settings scaffolding is framework-adjacent and stays where it is; the module structure is for **new** domain features.

## Key Patterns

- **Auth is Fortify-driven.** Login/register/password/2FA/verification routes and controllers are registered by Laravel Fortify — not in `routes/`. `app/Providers/FortifyServiceProvider.php` wires each Fortify view to an Inertia page and defines rate limiters. See [Security](security.md).
- **Shared validation via traits.** `app/Concerns/` holds rule traits consumed by both Fortify actions and Form Requests — define a rule once.
- **Inertia shared props.** `HandleInertiaRequests` shares `name`, `auth.user`, `sidebarOpen` to every page; read them via `usePage().props`.
- **Wayfinder bridges routes to TypeScript.** `resources/js/routes/` and `resources/js/actions/` are generated from PHP routes/controllers — import helpers from `@/routes/*` and `@/actions/*` instead of hardcoding URLs.
- **React Compiler is enabled**, so manual `useMemo`/`useCallback` is usually unnecessary.

## Data Flow (a typical request)

1. React issues an Inertia visit to a Laravel route.
2. The controller validates input (Form Request) and calls an application service.
3. The service loads data via a repository, invokes a domain method on the model, and persists.
4. The controller returns `Inertia::render('module/page', $props)` or a redirect.
5. React renders (or updates) the page component from the props.
